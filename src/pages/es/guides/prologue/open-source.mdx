# Código abierto (y fuente disponible) frente a propietario

En este sitio predico el software libre siempre que es posible. Es un tema muy importante que merece la pena explicar: ¿qué es la disponibilidad del código fuente, por qué la defiendo tanto y cuáles son sus ventajas e inconvenientes?

## ¿Qué es el software de código abierto?

La definición exacta de código abierto varía en función de las directrices de cada cual. Se trata de un tema importante con el que quiero familiarizar a mis lectores, pero que en general queda fuera del alcance de este sitio web. Si desea saber más sobre este tema, le recomiendo que empiece por la [Open Source Initiative](https://opensource.org/osd/) y la [Free Software Foundation](https://www.gnu.org/philosophy/free-sw.html#four-freedoms). Para los fines de este sitio, basta con saber que, en general, el software de código abierto es aquel que respeta la transparencia, la libertad del usuario y cumple algunos otros criterios técnicos.

**Este sitio web hace especial hincapié en la parte de "transparencia" del código abierto, concretamente en la "disponibilidad del código fuente "**. El software de código fuente disponible es aquel cuyo código fuente se ha publicado públicamente para que cualquiera pueda verlo. Esto puede incluir el software de código abierto, pero a veces el software que no es "de código abierto" por definición publicará su código fuente públicamente. En otras palabras: todo el software de código abierto es de código disponible, pero no todo el software de código disponible es de código abierto.

Hay varias razones por las que una organización o un desarrollador pueden publicar su código fuente, y de esas razones hay dos que me hacen poner mucho énfasis en el software disponible en código fuente. La primera razón es la confianza y la transparencia. Con el software de código fuente disponible, los expertos que saben leer e interpretar el código pueden confirmar que no hay nada poco ético en segundo plano, como la recopilación innecesaria de datos, o buscar posibles errores o vulnerabilidades, como una mala implementación de [cifrado](/guías/moderadamente-importante/cifrado). Una segunda razón estrechamente relacionada es que esas mismas personas pueden enviar sugerencias de mejora cuando encuentran esos problemas. Aunque dista mucho de ser una garantía, la disponibilidad del código fuente es una de las muchas herramientas que pueden ayudarnos a asegurarnos de que una aplicación o servicio hace lo que promete y no tiene debilidades evidentes. Consideremos la historia de [LastPass](https://www.wired.com/story/lastpass-breach-vaults-password-managers/), un gestor de contraseñas que no publicó su código fuente. Después de sufrir una brecha de datos en la que las bóvedas de los usuarios (bases de datos de contraseñas encriptadas) se vieron comprometidas, salió a la luz que LastPass estaba utilizando una mala implementación hash para las contraseñas maestras de los usuarios y no estaba encriptando las URL de los sitios almacenados en la bóveda, haciendo que las bóvedas fueran potencialmente fáciles de descifrar o ayudando a los ciberdelincuentes a saber en qué sitios tenías cuentas y crear fácilmente correos electrónicos de phishing convincentes para robar tus inicios de sesión en esos sitios. (Hablo más sobre la seguridad de las contraseñas en la página [Gestores de contraseñas](/guides/most-important/passwords)). Una vez más, mientras que la disponibilidad de la fuente por sí sola no puede garantizar que este tipo de problemas no existan, es muy probable que podría haber ayudado a LastPass a evitar esta vergonzosa y perjudicial (tanto para ellos como para sus clientes) revelación.

En el caso concreto del software de código abierto, una de las principales razones para poner el código a disposición de los usuarios es que éstos puedan modificarlo a su antojo y/o alojarlo de forma independiente para garantizar la seguridad de sus datos. No me centraré en este tema en particular en este sitio. Aunque el nivel exacto de conocimientos necesarios varía, en general el autoalojamiento o la modificación de software requiere un nivel relativamente alto de conocimientos técnicos sobre redes, programación y otros temas. Al principio de este sitio prometí que no esperaba que usted poseyera tales conocimientos técnicos, y lo decía en serio. Si estás interesado en avanzar en estos temas, hay una plétora de recursos disponibles en línea para ayudarte. Por ahora, sólo sepa que el software de código abierto de verdad -y no simplemente el software de código disponible- permite este tipo de control granular por parte de cualquiera. Incluso si no posees estas habilidades por ti mismo, te beneficiarás de este tipo de libertad si sigues algunas de las recomendaciones que enumero más adelante en este sitio (particularmente mis recomendaciones para un [navegador web](/guides/most-important/browser)). Un gran ejemplo que leí una vez decía **piensa en el código abierto como cocinar en casa y en el código propietario/cerrado como comer en un restaurante: en casa puedes ver cada ingrediente y tienes control total sobre cuáles añadir, excluir, sustituir o modificar. En el restaurante,** tu conocimiento de los ingredientes y el control sobre ellos está limitado a diversos grados, como sustituir un ingrediente o saber qué contiene la receta secreta.

## ¿Por qué es importante el código abierto?

Si mi énfasis se centra específicamente en el código fuente disponible, entonces ¿por qué me tomo la molestia de explicar el código abierto en su conjunto? Para empezar, debería explicar el **DRM**. DRM son las siglas de **Gestión de Derechos Digitales**, que es el término técnico para la tecnología antipiratería o contra el abuso de los derechos de autor. Permite a las empresas asegurarse de que estás utilizando una copia legítima de su software, juego o libro electrónico (u otros archivos digitales) en lugar de una versión pirateada, y también de que lo estás utilizando de acuerdo con las condiciones del servicio (por ejemplo, no alojando una sala de cine en tu casa). En cierto modo, hay una buena razón para que exista esta tecnología. Puede garantizar que los músicos cobren por su trabajo y que los productos populares tengan la oportunidad de continuar su éxito. Sin embargo, la DRM es propensa al abuso. Examinemos dos casos reales de DRM que han salido mal y cómo el código abierto puede beneficiar a todos.

Dos personas distintas han adquirido productos patentados diferentes: un [frigorífico](https://boingboing.net/2020/01/23/proprietary-carbon) y una [impresora](https://web.archive.org/web/20200117172133/https://twitter.com/ryandonsullivan/status/1218149470220632064). Esos productos vienen con accesorios adicionales que proporcionan fuentes de ingresos adicionales: filtros de agua y tinta de impresora, respectivamente. En un mercado tan competitivo como el actual, a menudo resulta más frugal (y legal) encontrar un tercero ajeno a la marca que ofrezca una pieza compatible por menos dinero que el producto del fabricante y que funcione igual de bien. Los fabricantes están empezando a responder haciendo que sus productos rechacen digitalmente el uso de accesorios de terceros. En el caso del frigorífico, el propietario se enteró cuando instaló un filtro de agua genérico de 19 dólares. El frigorífico utilizó DRM para confirmar que el filtro instalado no era un filtro oficial de GE y, por tanto, se negó a dispensar hielo o agua como antes. El filtro oficial cuesta 55 dólares. La historia de la impresora fue aún peor: el cliente se dio cuenta de que tenía una suscripción recurrente de 5 $/mes a "HP InstaInk". El cliente canceló esta suscripción, sin saber para qué servía. Después de cancelarla, la impresora se negó a imprimir nada, incluso con la tinta oficial de HP instalada.

Por un lado, se podría argumentar que se trata de una empresa que protege su inversión o propiedad intelectual, sobre todo porque muchos fabricantes venden el producto inicial con pérdidas o beneficios reducidos esperando recuperar el dinero en compras recurrentes. Pero **esto también marca una oscura tendencia en la que las empresas controlan todos los productos de nuestras vidas,** aplastando a la competencia. En el caso del frigorífico, si yo quisiera crear una empresa que vendiera un filtro que proporcionara agua más limpia, tendría que convencer a GE para que me contratara. Sin esta pesada DRM, puedo simplemente introducir mi producto en el mercado libre y dejar que el consumidor decida qué filtro le parece mejor. Pero este tipo de comportamiento anticompetitivo mantiene secuestrados los productos, poniendo límites irrazonables a lo que los consumidores pueden hacer con ellos y obligándoles a pagar precios exorbitantes sólo para obtener una funcionalidad básica de artículos por los que ya han pagado.

La era moderna está plagada de ejemplos de extralimitación de la DRM. En 2022, [BMW](https://www.theverge.com/2022/7/12/23204950/bmw-subscriptions-microtransactions-heated-seats-feature) empezó a exigir una suscripción para hacer uso de la función de asientos calefactados instalada en sus coches Pearson, la editorial que monopoliza esencialmente el espacio de los libros de texto académicos, abusó de su poder monopolístico para [subir los precios de los libros de texto](https://academicebookinvestigation.org/2021/11/25/outrage-as-pearson-increase-ebook-prices-by-500-in-one-week/) en 500 dólares en una sola semana. En 2009, Amazon retiró por completo las copias digitales de [1984](https://www.pcworld.com/article/519855/amazon_kindle_1984_lawsuit.html) de las bibliotecas de sus clientes de pago tras una disputa sobre derechos de autor. Los usuarios de Apple se indignaron en 2014 al despertarse una mañana y descubrir que se habían colocado copias del último álbum de U2 en sus [bibliotecas de iTunes](https://es.wikipedia.org/wiki/Songs_of_Innocence#Efectividad_y_reacci%C3%B3n) sin su consentimiento. No cabe duda de que hay que defenderse para luchar contra la piratería y garantizar que los productos no se utilicen con fines ilegales, pero todavía no he conocido a nadie que piense que estas historias no son un poco excesivas.

Hemos entrado en un nuevo mundo de conectividad en línea las 24 horas del día, los 7 días de la semana. Muchos coches tienen ahora sus propios módems integrados para conectarse a Internet desde cualquier lugar; nuestros electrodomésticos, como termostatos, bombillas, lavadoras y cafeteras, están constantemente conectados para controlarlos a distancia o por comodidad. A medida que la conectividad empieza a impregnar todos los elementos de nuestras vidas, es importante no sólo ser consciente de los [datos](https://www.forbes.com/sites/robertvamosi/2015/01/22/collecting-big-data-from-iot/) que envían esos dispositivos y de los [riesgos de seguridad](https://www.zdnet.com/article/iot-security-is-bad-its-time-to-take-a-different-approach/) de los mismos, sino también saber que ahora tienen la capacidad de aplicar las condiciones del servicio -que a menudo están sujetas a cambios en cualquier momento sin previo aviso- en cualquier momento y por cualquier motivo, como cuando las compañías eléctricas ajustaron a distancia [los termostatos inteligentes](https://www.usatoday.com/story/tech/2021/06/19/texas-power-companies-raising-smart-thermostat-temps-why/7754658002/) durante una ola de calor sin previo aviso para conservar energía y reducir la tensión en la red. Puede que tu coche no te denuncie por exceso de velocidad ahora mismo, pero tiene la capacidad de hacerlo y, en cualquier momento, el proveedor del servicio puede cambiar las reglas y empezar a informar de tus hábitos de exceso de velocidad al seguro y a [las fuerzas del orden](https://www.npr.org/sections/thetwo-way/2011/04/28/135809709/dutch-police-used-tomtoms-gps-data-to-target-speeders). **En el futuro, puede que tu coche sólo te permita repararlo con piezas del fabricante, o puede que decida que intentar repararlo en casa anula la garantía. Tomemos como ejemplo al conductor que se quedó [tirado](https://arstechnica.com/cars/2020/02/driver-stranded-after-connected-rental-car-cant-call-home/) cuando su coche de alquiler no pudo conectarse a la red.

**Los productos de código abierto protegen contra situaciones como ésta** porque están diseñados para proliferar. No puedes controlar a la competencia si pones el producto a libre disposición sin restricciones. No se puede impedir que usuarios anónimos lo compartan y modifiquen. Incluso si se intenta imponer la DRM, el código fuente puede modificarse para eliminar esa imposición. Un frigorífico de código abierto, por ejemplo, podría modificarse fácilmente para eliminar los bloqueos digitales que exigen los filtros del fabricante. Protege a los consumidores de los monopolios anticompetitivos que fijan precios abusivos, recopilan demasiados datos, no aplican la seguridad adecuada y ponen restricciones poco razonables a los usuarios por los productos que ya han pagado.

A pesar de todos mis elogios en esta página, es importante señalar que los productos de código abierto no son automáticamente [garantizados](https://blog.thenewoil.org/open-source-does-not-always-equal-safe) más seguros o privados. Sin embargo, con el código disponible libremente hay menos margen para el abuso y más oportunidades para que una comunidad activa e implicada ayude a mejorar el producto.
